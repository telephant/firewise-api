# DCA Recurring Investment Design

**Date:** 2026-05-01
**Status:** Approved

---

## Overview

Add a Dollar-Cost Averaging (DCA / 定投) feature to the Portfolio module. Users define recurring investment plans (e.g. buy $500 of QQQ every month). When a plan comes due, the backend generates a **pending confirmation record** — the user reviews it, fills in the actual price and shares, and confirms to create a real trade. No auto-execution, always user-confirmed.

## Data Model

### `dca_plans` table

Stores the user's recurring investment plan configuration.

```sql
CREATE TABLE dca_plans (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  portfolio_id    UUID NOT NULL REFERENCES portfolios(id) ON DELETE CASCADE,
  ticker          TEXT NOT NULL,
  market          TEXT NOT NULL CHECK (market IN ('US', 'SGX', 'HK', 'CN')),
  currency        TEXT NOT NULL,
  frequency       TEXT NOT NULL CHECK (frequency IN ('weekly', 'biweekly', 'monthly', 'quarterly', 'yearly')),
  mode            TEXT NOT NULL CHECK (mode IN ('amount', 'shares')),
  amount          DECIMAL(18,8),   -- set when mode = 'amount', NULL otherwise
  shares          DECIMAL(18,8),   -- set when mode = 'shares', NULL otherwise
  next_run_date   DATE NOT NULL,
  last_run_date   DATE,
  is_active       BOOLEAN NOT NULL DEFAULT TRUE,
  notes           TEXT,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**RLS:** Access via `portfolio_id → portfolios.belong_id` — no redundant `belong_id` column needed.

### `dca_pending` table

One row per scheduled occurrence, generated by the cron task when a plan comes due. Stays until user confirms or skips.

```sql
CREATE TABLE dca_pending (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  dca_plan_id       UUID NOT NULL REFERENCES dca_plans(id) ON DELETE CASCADE,
  portfolio_id      UUID NOT NULL REFERENCES portfolios(id) ON DELETE CASCADE,
  ticker            TEXT NOT NULL,
  market            TEXT NOT NULL,
  currency          TEXT NOT NULL,
  scheduled_date    DATE NOT NULL,
  mode              TEXT NOT NULL CHECK (mode IN ('amount', 'shares')),
  amount            DECIMAL(18,8),          -- plan amount (mode=amount)
  shares            DECIMAL(18,8),          -- plan shares (mode=shares)
  suggested_price   DECIMAL(18,8),          -- latest price fetched at generation time
  suggested_shares  DECIMAL(18,8),          -- amount / suggested_price (mode=amount)
  status            TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'confirmed', 'skipped')),
  confirmed_price   DECIMAL(18,8),          -- filled by user on confirm
  confirmed_shares  DECIMAL(18,8),          -- filled by user on confirm
  trade_id          UUID REFERENCES trades(id) ON DELETE SET NULL,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**RLS:** Access via `portfolio_id → portfolios.belong_id`.

## Backend Architecture

### New files

**`src/controllers/dca.controller.ts`**

Functions:
- `listPlans(req, res)` — GET all plans for current family context (joins portfolios via belong_id)
- `createPlan(req, res)` — validates mode/amount/shares, sets next_run_date = start_date from body
- `updatePlan(req, res)` — update any plan fields; recalculates next_run_date if frequency changes
- `deletePlan(req, res)` — hard delete (cascades to dca_pending)
- `listPending(req, res)` — GET all pending records (status=pending) for current family context
- `confirmPending(req, res)` — body: `{ confirmed_price, confirmed_shares }`; creates trade via tradeApi logic; sets status=confirmed, trade_id; returns created trade
- `skipPending(req, res)` — sets status=skipped

**`src/routes/dca.routes.ts`**

```
GET    /fire/dca/plans              → listPlans
POST   /fire/dca/plans              → createPlan
PUT    /fire/dca/plans/:id          → updatePlan
DELETE /fire/dca/plans/:id          → deletePlan
GET    /fire/dca/pending            → listPending
POST   /fire/dca/pending/:id/confirm → confirmPending
POST   /fire/dca/pending/:id/skip   → skipPending
```

Registered in `src/routes/index.ts`.

**`tasks/process-dca.task.ts`**

Runs daily (via external cron or manual trigger). Logic:
1. Query all `dca_plans` where `is_active = true` AND `next_run_date <= TODAY`
2. For each plan, fetch latest stock price (reuse existing `price_cache` / stockPriceApi logic)
3. Insert row into `dca_pending`:
   - Copy ticker/market/currency/mode/amount/shares from plan
   - `scheduled_date` = plan's `next_run_date`
   - `suggested_price` = fetched price
   - `suggested_shares` = `amount / suggested_price` (only when mode=amount, rounded to 6dp)
4. Update plan: `last_run_date = next_run_date`, advance `next_run_date` by frequency
5. Log counts

Invoked via: `npx ts-node tasks/index.ts process-dca`

Also exposed as: `POST /fire/dca/process` (auth-protected, for testing / manual trigger)

### next_run_date advancement

| frequency | advance by |
|-----------|-----------|
| weekly | +7 days |
| biweekly | +14 days |
| monthly | +1 month |
| quarterly | +3 months |
| yearly | +1 year |

## Confirm Flow (UX detail)

```
Pending card shows:
  QQQ · US
  Scheduled: 2026-05-01
  Est. Price: $480.20          ← suggested_price from cron
  Shares: 1.041                ← suggested_shares (mode=amount, $500 / $480.20)
  ┌─────────────────┐  ┌──────────────┐
  │ Price  [480.20] │  │ Shares [1.04]│   ← both editable
  └─────────────────┘  └──────────────┘
  [Confirm]  [Skip]
```

On Confirm:
- POST `/fire/dca/pending/:id/confirm` with `{ confirmed_price, confirmed_shares }`
- Backend creates trade: `{ ticker, market, type:'buy', shares:confirmed_shares, price:confirmed_price, currency, date:scheduled_date, portfolio_id }`
- Returns created trade
- Frontend removes card from pending list, refreshes portfolio holdings if on portfolio page

## Migration

Add to `supabase/migrations/001_migration.sql` (append after existing tables):

```sql
-- DCA Plans
CREATE TABLE IF NOT EXISTS dca_plans ( ... );

-- DCA Pending
CREATE TABLE IF NOT EXISTS dca_pending ( ... );

-- RLS policies (access via portfolio join)
ALTER TABLE dca_plans ENABLE ROW LEVEL SECURITY;
ALTER TABLE dca_pending ENABLE ROW LEVEL SECURITY;

CREATE POLICY "dca_plans_access" ON dca_plans
  USING (portfolio_id IN (
    SELECT id FROM portfolios
    WHERE belong_id = auth.uid()
       OR belong_id IN (SELECT family_id FROM family_members WHERE user_id = auth.uid())
  ));

CREATE POLICY "dca_pending_access" ON dca_pending
  USING (portfolio_id IN (
    SELECT id FROM portfolios
    WHERE belong_id = auth.uid()
       OR belong_id IN (SELECT family_id FROM family_members WHERE user_id = auth.uid())
  ));
```
